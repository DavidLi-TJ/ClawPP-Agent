## 关键事实：飞书应该怎么接
### 自定义机器人
适合“群内推送通知”。
- 通过 webhook 发送消息。
- 不需要 `app_secret`。
- 有关键词、IP 白名单、签名校验等安全配置。
- 只能在当前群聊内使用，不能用于完整的双向消息机器人。

### 机器人应用 / 自建应用
适合“接收消息、解析用户输入、回复用户、拿 open_id、做双向交互”。
- 需要 `app_id` + `app_secret`。
- 需要换取 `tenant_access_token`。
- 需要订阅消息事件，配置回调地址。
- 需要按官方安全要求做签名校验、事件验签、消息体解析。

结论：如果要做“飞书里和 Agent 对话”的完整能力，必须走自建应用路线，而不是只靠自定义机器人 webhook。

## GUI 现代化方向
参考 OpenAkita 的思路，但不照搬技术栈，只吸收结构和体验：

- 左侧固定导航栏：聊天、会话、技能、记忆、外部平台、设置、日志、关于。
- 右侧主内容区：使用卡片化布局，不再堆平铺表单。
- 顶部状态条：显示连接状态、模型名、会话状态、token 使用量。
- 信息面板化：
  - Agent 状态
  - Memory 状态
  - Skill 安装/启用状态
  - 外部平台状态
- 增加更明显的视觉层次：标题、说明文字、分区卡片、状态徽章、空状态页。
- 所有按钮、标签、菜单、错误提示统一中文。
- 增加“引导式设置”体验，减少一次性暴露所有复杂选项。

## 对照 spec 还缺的主要功能
### 优先级 1：飞书完整双向能力
- 用自建应用方式重做飞书接入。
- 支持事件订阅、验签、消息接收、用户回复、群聊/私聊区分。
- 支持发送富文本或卡片消息。
- 支持 appsecret 配置和自动刷新 token。

### 优先级 2：多模型提供商
- 解决当前 provider 只覆盖 OpenAI 兼容接口的问题。
- 加入至少一个国内模型适配器和一个本地模型适配器。
- 做统一的 provider 配置页和失败回退策略。
- 针对 429 / 限流错误做重试、退避和友好提示。

### 优先级 3：技能系统体验
- 技能市场、已安装技能、技能配置、启停、依赖检查。
- 技能详情页中文化。
- 技能加载失败时给出可读错误。

### 优先级 4：记忆系统体验
- 记忆面板可视化。
- token 压缩阈值、摘要策略、历史查看。
- 记忆压缩过程和结果可追踪。

### 优先级 5：整体产品化
- 统一主题样式。
- 空状态、加载态、错误态。
- 日志和可观测性。
- 更完整的设置中心。
- 更清晰的“聊天 / 设置 / 技能 / 外部平台”信息流。

## 设计原则
- 继续保留 Qt/C++，不迁移框架。
- 先做结构，再做皮肤。
- 先做能用，再做精致。
- 统一中文术语，不保留半套英文半套中文。
- 任何外部平台接入都按官方安全模型实现，不走“看起来能用”的临时方案。

## 当前风险
- 飞书如果用错成“自定义机器人”方案，会缺少双向消息能力。
- GUI 如果只做颜色和圆角，不改布局结构，效果不会像 OpenAkita 那样完整。
- 多模型 provider 如果只补壳不补重试和限流处理，429 还会频繁暴露给用户。
- 中文化如果不统一消息源，后面会继续出现中英混杂。

## 建议下一步
1. 先把飞书改成自建应用路线，并补充 `app_secret`、验签、事件订阅和回复发送。
2. 再重构主界面导航与设置页，按 OpenAkita 的信息分区方式做现代化布局。
3. 然后补 Provider 层，优先解决限流和多模型支持。
4. 最后再收敛技能市场、记忆管理和日志中心。

## 备注
- 当前计划是面向 Qt Widgets 的落地方案。
- 如果后续要更接近 OpenAkita 的信息密度和视觉风格，可以继续做更强的卡片化、侧栏化和仪表盘化改造。
